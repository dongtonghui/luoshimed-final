# 静态网站项目架构思考框架

> 基于骆氏健康官网完整建设周期的深度复盘

---

## 一、项目背景回顾

### 原始诉求
- 企业官网（医疗器械/健康领域）
- 10个页面：首页、关于、产品、案例、研发、诊所、联系、下载、新闻、产品详情
- 需要支持后续内容自主更新
- 无后端开发资源

### 建设历程
1. **V1.0** - 纯静态HTML页面（内容硬编码）
2. **V2.0** - 动态配置化改造（内容配置化）

---

## 二、核心思考框架：五层架构法

```
┌─────────────────────────────────────────────────────────┐
│  Layer 5: 内容层 (Content)                               │
│  - 业务文案、产品信息、案例数据                            │
│  - 图片、视频、文档资源                                   │
│  - 配置化存储 (JSON/CMS)                                 │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  Layer 4: 信息架构层 (Information Architecture)           │
│  - 页面结构、导航关系、URL设计                            │
│  - 内容分类、标签体系、搜索逻辑                            │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  Layer 3: 表现层 (Presentation)                          │
│  - HTML结构、CSS样式、交互逻辑                            │
│  - 响应式适配、性能优化                                   │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  Layer 2: 交付层 (Delivery)                              │
│  - 托管方案、CDN加速、HTTPS                               │
│  - SEO优化、访问统计、表单处理                             │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  Layer 1: 运营层 (Operations)                            │
│  - 内容更新流程、版本管理、备份策略                         │
│  - 数据分析、迭代优化                                     │
└─────────────────────────────────────────────────────────┘
```

## 三、各层关键决策点

### Layer 5: 内容层 - "什么信息值得被展示"

#### 决策1: 内容所有权划分
| 内容类型 | 更新频率 | 维护者 | 存储方式 |
|---------|---------|--------|---------|
| 公司基本信息 | 低 | 运营 | JSON配置 |
| 产品信息 | 中 | 产品 | JSON配置 |
| 案例/新闻 | 高 | 市场 | JSON配置 |
| 页面结构 | 极低 | 开发 | HTML模板 |

#### 决策2: 内容结构化程度
```
误区：所有内容都硬编码在HTML中
      ↓
改进：将"会变化的内容"提取为配置
      ↓
进阶：建立内容模型（Schema）
```

**本次项目的内容模型设计：**
```
Product: { id, name, description, image, specs[], certifications[] }
Case: { id, type, name, description, metrics[], image }
News: { id, category, title, summary, date, image }
```

---

### Layer 4: 信息架构层 - "信息如何被组织"

#### 决策1: 导航架构设计

**原始思路（按部门划分）：**
- 关于我们、产品中心、合作案例...

**优化思路（按用户场景）：**
- 我想了解你们（关于、案例、研发）
- 我想购买产品（产品、联系）
- 我想合作加盟（诊所、联系）

**最终采用：部门+场景混合架构**
```
关于骆氏 → 品牌认知
产品中心 → 产品探索
合作案例 → 信任背书
研发实力 → 专业背书
诊所业务 → 加盟合作 ← 明确的CTA
新闻中心 → 动态更新
联系我们 → 转化入口
```

#### 决策2: 页面层级深度
```
原则：核心转化路径不超过3层

首页 (1层)
  ├── 产品详情页 (2层) ← 直接转化
  ├── 联系表单 (2层) ← 直接转化
  └── 诊所业务 (2层)
        └── 加盟表单 (3层)
```

#### 决策3: URL设计策略
```
/                       # 首页
/about.html             # 关于我们
/products.html          # 产品列表
/product-detail.html?id=1  # 产品详情（带参数）
```

**反思：** 产品详情页使用查询参数而非 RESTful URL，是权衡后的选择：
- 优点：单文件支持多产品，维护简单
- 缺点：SEO略逊，但产品量不大时可接受

---

### Layer 3: 表现层 - "如何优雅地呈现"

#### 决策1: 技术栈选择
| 维度 | 选择 | 理由 |
|-----|------|-----|
| 构建工具 | 无 | 避免构建复杂度，直接编辑HTML |
| CSS框架 | Tailwind CSS CDN | 快速样式，无需构建 |
| JS框架 | 无 | 纯展示型网站，Vanilla JS足够 |
| 配置格式 | JSON | 原生支持，无需解析库 |

#### 决策2: 组件复用策略
```
识别可复用组件：
- 导航栏（所有页面）
- 页脚（所有页面）
- 页面横幅（多个页面）
- 联系表单卡片（首页+联系页）

策略：
- 不抽取为JS组件（避免构建工具）
- 使用CSS类名约定保持一致性
- 通过配置控制内容差异
```

#### 决策3: 响应式断点
```css
/* 移动端优先 */
default:  < 768px    /* 手机 */
md:       >= 768px   /* 平板 */
lg:       >= 1024px  /* 小桌面 */
xl:       >= 1280px  /* 标准桌面 */
```

---

### Layer 4: 交付层 - "如何可靠地交付"

#### 决策1: 托管方案
- 选择：GitHub Pages / Cloudflare Pages / Vercel
- 理由：免费、自动部署、自带CDN、HTTPS

#### 决策2: 表单处理
```
问题：静态网站如何收集表单数据？

方案对比：
A. 自建后端 - 成本高
B. Formspree / Getform - 第三方依赖
C. 直接mailto - 体验差

本次选择：预留表单结构，实际部署时接入客户自有系统
```

#### 决策3: SEO基础
```html
<!-- 每个页面独立的 SEO 配置 -->
<title>页面特定标题 | 品牌名</title>
<meta name="description" content="页面特定描述">

<!-- Open Graph -->
<meta property="og:title" content="...">
<meta property="og:description" content="...">
<meta property="og:image" content="...">
```

---

### Layer 5: 运营层 - "如何持续迭代"

#### 决策1: 内容更新流程
```
传统流程：
运营提需求 → 开发修改HTML → 测试 → 发布

配置化流程：
运营修改JSON → 自动部署 → 完成

关键：让运营能自助完成80%的文案更新
```

#### 决策2: 版本管理策略
```
Git 分支策略：
- main: 生产环境
- develop: 开发测试
- feature/*: 功能分支

配置文件版本：
- 配置与代码同仓库，保证版本一致性
- 重要变更打 Tag
```

## 四、关键设计权衡

### 权衡1: 配置化 vs 硬编码

| 场景 | 选择 | 理由 |
|-----|------|-----|
| 公司名称 | 配置化 | 可能变更，多处使用 |
| 产品列表 | 配置化 | 经常增删，数据结构复杂 |
| 页面布局 | 硬编码 | 变更频率低，涉及样式 |
| 导航结构 | 配置化 | 可能调整，影响全局 |

### 权衡2: 单文件 vs 多文件
```
产品详情页设计：

方案A: product-1.html, product-2.html...
   - 优点：SEO友好，独立页面
   - 缺点：文件多，维护难

方案B: product-detail.html?id=1
   - 优点：单文件，配置驱动
   - 缺点：SEO略差

选择：方案B（产品量小，追求维护效率）
```

### 权衡3: 动态渲染 vs 静态生成
```
方案A: 构建时生成（SSG）
   - 需要构建工具
   - 复杂度高

方案B: 运行时渲染（CSR）
   - 简单直接
   - SEO需额外处理

选择：方案B（无需构建，配置即改即生效）
```

## 五、反模式与避坑指南

### ❌ 反模式1: 过度工程化
```
误区：
- 为展示型网站引入 React/Vue
- 配置系统过于复杂（数据库+管理后台）

正解：
- 技术选型匹配项目复杂度
- 简单的JSON配置足够时，不引入CMS
```

### ❌ 反模式2: 配置泛滥
```
误区：
- 所有内容都配置化
- 包括颜色、间距等样式

正解：
- 只配置"业务内容"
- 样式控制在CSS中
```

### ❌ 反模式3: 忽视性能
```
误区：
- 大量JSON数据同步加载
- 未压缩的资源

正解：
- 配置按需加载（如需要）
- 图片懒加载
- 资源压缩
```

## 六、总结：静态网站项目的决策树

```
开始新项目
    │
    ├─ 内容更新频率？
    │   ├─ 极少（年更）→ 纯静态HTML
    │   └─ 中等（月更）→ JSON配置化 ← 本项目选择
    │       └─ 更新非常频繁（周更）→ 考虑 Headless CMS
    │
    ├─ 页面数量？
    │   ├─ 少于10页 → 单文件配置足够
    │   └─ 多于50页 → 考虑分页配置或CMS
    │
    ├─ 是否需要后端功能？
    │   ├─ 仅表单收集 → 静态+第三方服务
    │   └─ 用户系统/支付 → 必须后端
    │
    └─ 团队技术能力？
        ├─ 纯运营人员 → 可视化CMS
        └─ 有技术人员 → JSON配置更灵活
```

## 七、本次项目的核心经验

### 做得好的地方 ✅
1. **分层清晰** - 内容/架构/表现分离
2. **配置粒度适中** - 不过度设计
3. **维护友好** - 运营可自行修改80%内容
4. **零构建** - 降低技术门槛

### 可以改进的地方 ⚠️
1. **初期未考虑配置化** - 后期改造成本
2. **缺少可视化预览** - 修改配置后需刷新看效果
3. **表单处理不完整** - 需接入实际业务系统

### 下次会这样做 🚀
1. **项目启动即设计配置Schema** - 避免返工
2. **组件化思维** - 即使不用React，也要有组件概念
3. **文档先行** - CONFIG_README与开发同步进行
