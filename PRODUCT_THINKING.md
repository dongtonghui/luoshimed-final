# 网站建设的业务思考框架

> 从"做网站"到"做业务增长工具"

---

## 一、重新定义问题

### 原始需求 vs 真实需求

| 表面需求 | 真实需求 | 洞察 |
|---------|---------|-----|
| "做一个企业官网" | "让潜在客户了解我们并产生联系" | 网站是销售工具 |
| "展示公司产品" | "建立信任，降低决策门槛" | 医疗器械需要强背书 |
| "可以修改文案" | "快速响应市场变化" | 业务敏捷性 |

### 核心问题转化
```
❌ "如何做一个好看的网站？"
   ↓
✅ "如何设计一个能获客的数字化店面？"
   ↓
🎯 "如何让不同角色的访客快速找到他们关心的信息并采取行动？"
```

## 二、用户旅程地图

### 访客类型识别（B2B医疗器械）

```
类型1: 医疗机构采购决策者
  关心点：资质认证、临床效果、同行案例、售后支持
  路径：首页 → 资质 → 案例 → 联系
  
类型2: 诊所加盟意向者
  关心点：加盟政策、整店输出、回本周期、培训支持
  路径：首页 → 诊所业务 → 案例 → 表单
  
类型3: 经销商/渠道商
  关心点：产品优势、利润空间、区域保护、市场支持
  路径：首页 → 产品 → 合作政策 → 联系
  
类型4: 普通患者/家属
  关心点：品牌可信度、治疗效果、体验店位置
  路径：首页 → 关于 → 诊所（找门店）
```

### 关键决策点设计

```
每个页面都要回答：
1. 用户现在在哪？（位置感知）
2. 他们想要什么？（需求匹配）
3. 下一步该做什么？（明确CTA）

示例 - 产品页设计：
┌─────────────────────────────────────┐
│  Banner: 核心产品系列               │  ← 我在产品中心
│  副标题: 医用/家用/智能三大系列      │  ← 满足什么需求
├─────────────────────────────────────┤
│  筛选标签: [全部][医用][家用][智能] │  ← 快速找到目标
│                                     │
│  产品卡片 x 6                       │  ← 详细信息
│  - 图片                             │
│  - 名称                             │
│  - 简介                             │
│  - 注册证号 ← 医疗器械必需要素       │
│  - 查看详情 →                       │  ← 明确CTA
├─────────────────────────────────────┤
│  CTA区块: 需要定制化方案？ → 联系我们 │  ← 捕捉未满足需求
└─────────────────────────────────────┘
```

## 三、信息架构的业务逻辑

### 导航设计的转化逻辑

```
不是：我们要展示什么
而是：访客想找到什么

原导航思路：
关于我们 | 产品中心 | 合作案例 | 研发实力 | 诊所业务 | 新闻中心 | 联系我们

优化后的转化逻辑：
【了解】           【购买/合作】        【行动】
关于我们           产品中心           联系我们
合作案例 → 信任    诊所业务 → 加盟     表单转化
研发实力 → 专业    
```

### 页面优先级排序

```
高优先级（核心转化）：
- 首页：流量入口，分流引导
- 诊所业务：加盟是核心商业模式
- 联系我们：最终转化页面

中优先级（信任建立）：
- 合作案例：第三方背书
- 研发实力：专业度证明
- 产品中心：需求匹配

低优先级（辅助信息）：
- 关于我们：基础信息
- 新闻中心：SEO和活跃度
- 下载中心：售后支持
```

## 四、内容策略

### 内容生产的业务视角

```
每块内容都要回答三个问题：

1. 商业价值：带来什么业务结果？
   - 案例页 → 建立信任 → 促进咨询
   
2. 用户价值：解决了用户什么疑问？
   - 资质认证 → 解决"是否合法"的担忧
   
3. 差异化：与竞品有什么不同？
   - "百年传承+现代科技"的定位
```

### 本次项目的内容框架

```
品牌故事层（情感连接）
├── 百年传承的品牌故事
├── 创始人的使命感
└── "科技正脊找骆氏"的口号

专业信任层（理性说服）

├── 20+医疗机构合作
├── 92%临床有效率
├── ISO 13485认证

产品方案层（解决方案）
├── 医用系列（B2B）

商业模式层（合作路径）
├── 产品采购
├── 诊所加盟（整店输出）
└── 区域分销
```

## 五、技术决策的业务考量

### 为什么选择配置化？

```
业务场景：医疗器械行业
特点：
- 产品信息频繁更新（新品上市、规格调整）
- 案例需要持续积累（增加可信度）
- 市场活动需要快速响应（限时政策）

解决方案对比：

纯静态：
优点：简单
缺点：改文案需开发介入，响应慢

配置化（本项目选择）：
优点：运营自主更新，分钟级响应
缺点：需一定的JSON编辑能力

CMS系统：
优点：可视化编辑
缺点：增加复杂度，需要学习成本

决策：选择配置化，在灵活性和复杂度间取得平衡
```

### 性能与业务的平衡

```
业务需求：
- 首屏打开要快（降低跳出率）
- 图片要清晰（展示产品细节）
- 手机访问要流畅（决策者多用手机）

技术决策：
- 图片使用 Unsplash + 压缩参数
- CDN引入Tailwind（缓存优势）
- 无构建工具（简化部署）
```

## 六、增长视角的网站设计

### 关键指标定义

```
不仅仅是PV/UV，而是：

获取阶段：
- 自然搜索流量占比
- 品牌词搜索量

激活阶段：
- 首页跳出率
- 平均停留时长
- 页面深度

转化阶段：
- 咨询表单提交数
- 电话拨打次数
- 诊所加盟线索数

留存阶段：
- 回头客比例
- 案例页二次访问
```

## 七、维护与迭代的业务逻辑

### 内容更新的业务节奏

```
每日：
- 查看表单提交（及时响应线索）

每周：
- 新增案例（保持内容新鲜度）
- 新闻更新（行业动态/公司动态）

每月：
- 产品信息更新
- 页面文案优化（基于数据）

每季：
- 整体内容策略 review
- 竞品网站对标分析
```

### 本次项目的可维护性设计

```
考虑点1: 谁来更新？
→ 运营人员
→ 需提供 CONFIG_README.md

考虑点2: 更新频率？
→ 中等（每月数次）
→ JSON配置足够，无需CMS

考虑点3: 更新风险？
→ Git版本控制，可回滚
→ 修改前备份

考虑点4: 更新后验证？
→ 本地测试服务器
→ 简单测试脚本
```

## 八、总结：从交付网站到交付价值

### 本次项目的价值交付

```
不只是交付了10个HTML页面，而是交付了：

1. 获客工具
   - 清晰的转化路径
   - 可信的内容呈现
   - 便捷的联系方式

2. 品牌资产
   - 统一的品牌表达
   - 可积累的案例库
   - 专业的内容框架

3. 运营基础设施
   - 可自主更新的配置系统
   - 标准化的发布流程
   - 可传承的知识文档
```

### 关键成功因素

```
✅ 深度理解业务（医疗器械B2B+B2C）
✅ 用户视角设计（不同角色的转化路径）
✅ 内容策略先行（不急于写代码）
✅ 可维护性考量（配置化设计）
✅ 适度技术选型（匹配团队能力）
```
